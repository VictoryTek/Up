# Up 2.0.1

Released: 2026-05-12

## Bug Fixes

### Nix Build: up-daemon Now Correctly Compiled and Installed

Fixed two compounding bugs that caused the Nix `installPhase` to fail with:

```
install: cannot stat 'target/release/up-daemon': No such file or directory
```

**Root cause 1 — daemon never compiled:**  
`cargoBuildHook` was invoked without `--workspace`, so only the root `up` crate was compiled. The `daemon/` workspace member was never built, producing no `up-daemon` binary.

**Fix:** Added `cargoBuildFlags = [ "--workspace" ]` to the `buildRustPackage` derivation in `flake.nix`. This causes `cargoBuildHook` to pass `--workspace` to cargo, compiling all workspace members, and causes `cargoInstallHook` to install all workspace binaries to `$out/bin/`.

**Root cause 2 — wrong install path:**  
The `postInstall` script referenced `target/release/up-daemon`, which never exists in a nixpkgs build because `cargoBuildHook` always passes `--target <arch-triple>`, placing artifacts under `target/<arch-triple>/release/`.

**Fix:** Updated `postInstall` to move `up-daemon` from the `$out/bin/` location where `cargoInstallHook` places it to the correct `$out/libexec/up-daemon` destination.

Reproduced on: NixOS 25.11, nixpkgs 8fd9daa3, Up 2.0.0.

### NixOS Flake Update: nixos-rebuild Now Passes --refresh

Fixed a bug where `nixos-rebuild switch` could silently use stale cached content after `nix flake update` had already written a new `flake.lock`.

**Root cause:** Nix maintains a TTL-based download/tarball cache (default TTL: 3600 s). After `nix flake update` writes new locked revisions, the subsequent `nixos-rebuild switch` evaluation step could still hit this cache and re-use the previously downloaded tarballs — meaning the rebuild referenced the old closure rather than the inputs recorded in the freshly updated lock file. This was most visible with rolling inputs such as `nixos-unstable`, force-pushed branches, or channel tarballs mutated in place.

**Fix:** Added `--refresh` to the `nixos-rebuild switch` invocation inside the flake-based NixOS update path in `src/backends/nix.rs`. The `--refresh` flag instructs Nix to treat all previously downloaded files as out-of-date, bypassing the TTL-based cache and re-fetching inputs from their canonical sources.

**Why `--refresh` on `nixos-rebuild switch` and not on `nix flake update`:** `nix flake update` writes the lock file directly from the registry or network and does not consult the download cache for its final output. The download cache is consumed during evaluation and building — i.e. during `nixos-rebuild switch` — so that is where `--refresh` must be placed.

Reproduced on: NixOS 25.11, nixos-unstable input with sub-TTL re-run, Up 2.0.0.
